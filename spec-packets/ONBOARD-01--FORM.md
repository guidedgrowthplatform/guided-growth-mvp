### AI Context Block

SCREEN: Profile Setup (Step 1)
STATE: User just heard the welcome + presence ask. They're engaged and present.
*** THIS IS THE FIRST REAL-TIME AGENT SCREEN ***
Unlike screens before this (which were Vapi one-way broadcasts), this screen activates a full Vapi agent conversation. The AI speaks LIVE through real-time TTS, then listens to user voice via Soniox STT (with Gladia auto-fallback), then responds again. Everything the AI says on this screen is generated by GPT and spoken by Cartesia in the cloned voice.
FLOW:
1. Screen loads. Vapi agent connection opens automatically.
2. Agent speaks first (introduction): asks for name, age, gender, referral.
3. User responds by voice (or taps form fields).
4. Agent responds with the user's NAME via live TTS: 'Great to meet you, [Name].'
   This is the key test - the AI says the user's actual name back to them in the cloned voice.
5. If user gave partial info, agent asks for remaining fields only.
6. Form auto-fills in real-time as user speaks.
BEHAVIOR: Accept voice or taps. Auto-fill form fields as voice is parsed. If user gives partial info (just name), ask for remaining fields only - never re-ask what you already got. Confirm with a warm greeting using their name.
PARSING: 'Call me Sam' = name Sam. 'Twenty-five' = age 25. 'Guy/man/boy' = Male. 'Girl/woman/lady' = Female. 'Non-binary' = Other. 'TikTok/IG' = Social media. 'A friend' = Friend. 'Google' = Website.
DO NOT: Re-ask fields already captured. Push on gender if declined. Make referral source feel mandatory.
NEXT: All fields filled -> ONBOARD-FORK (experience fork).

## Applicable UX rules

### UX-02 - Mic stays listening once granted

**Applies to:** All LLM-active screens (ONBOARD-01 onwards, all check-ins, voice conversations)

Once the user grants microphone permission and the mic is ON, it stays listening throughout an active voice session. The user should NEVER have to tap the mic again to continue a conversation. Mic remains active throughout the entire session. The only time a user taps the mic is to turn it back ON after they (or the system) turned it OFF.

### UX-03 - Voice-driven navigation (no tap to advance)

**Applies to:** All LLM-active screens in voice mode

In voice mode, the user should NEVER have to press a button to advance to the next screen. The conversation IS the navigation. When the agent detects a step is complete, the LLM calls navigate_next, the frontend transitions, the new screen's context is fetched, and the conversation continues. Navigation logic stays in the agent's system prompt, not hardcoded per screen.

### UX-09 - Both voice and text/tap always work

**Applies to:** All screens

Every action in the app can be done via voice OR via text/tap. Voice is the preferred experience but never the only path. If mic is denied or voice isn't working, the user can still complete every flow with taps and typing.

### UX-13 - Coaching response brevity

**Applies to:** All LLM-active screens, all coaching responses

Coaching responses are short by default. Morning check-in: 1-2 sentences. Evening check-in: 2-3 sentences. Voice conversations: 4-5 sentences max. No speeches. The exception is the welcome MP3 (~60s) and milestone moments.

### UX-14 - Coaching style: Warm & Thoughtful (MVP)

**Applies to:** All LLM-active screens, SETTINGS

MVP uses a single coaching style: Warm & Thoughtful. Other styles (Honest & Direct, Calm & Reflective) are post-MVP. Do NOT show coaching style selector in MVP settings. Default and only option.

### UX-17 - AI button - voice/text mode toggle

**Applies to:** All screens with AI presence

AI button has two modes: voice (AI speaks via TTS) or text (AI writes only). User picks default at VOICE-PREFERENCE. User can toggle anytime. Voice -> text flip cuts AI audio immediately (no finish-the-sentence). When in text mode AND chat overlay is closed, AI text appears as subtitles overlay (user can minimize).

### UX-24 - Orb has two independent toggles; form/voice variant rendering keyed to combined state

**Applies to:** All screens with form/voice variant frames (ONBOARD-01, ONBOARD-FORK, MIC-PERMISSION, future check-in screens)

The orb in the bottom bar has two halves, each tappable independently. They never fire together. RIGHT HALF (mic button) controls mic state: active (blue, listening) -> user tap -> off (sticky) -> tap again -> active. Independently, 8 seconds of silence drops mic to idle (gray, still listening but quiet) per UX-16. Any user interaction reactivates from idle. LEFT HALF (AI button) controls AI output mode: voice (AI speaks via TTS) vs text (AI writes only, subtitle overlay appears, see UX-25). User tap flips. Voice -> text flip cuts AI audio immediately per UX-17. VARIANT RENDERING: a screen renders its --VOICE variant when BOTH (a) the AI button is in voice mode AND (b) the mic is active (not off, not idle). When either toggle is off / idle / gray, the screen renders the --FORM variant. Same screen ID, same data captured, same tool calls; only rendering differs. To get per-half independent tapping in the prototype, Timothy needs to split the orb component into two distinct interactive layers. Until that lands, the prototype's 'dual button' tap is a one-target approximation. Cross-refs: UX-05, UX-16, UX-17, UX-18, UX-25.
