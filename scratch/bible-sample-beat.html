Build watch-outs

6 · for the packet, not for you

Where the sample hit a judgment call or a gap. I resolve these in the build packet, they are here for transparency.

  1. Shared screenId across two beats. category and category-women both use ONBOARD-BEGINNER-01. beatId is the only unique key; the gender-variant code rule disambiguates. The alias-check must permit two beatIds on one screenId while keeping the rest unique.
  2. Persistence table + column unknown. The source and docs do not name where the category write lands. Left conceptual, flagged for app-reconcile, not invented.
  3. submit_category enum not locked in source. The 8 labels are inferred from the goals-* beats; the exact enum + the custom-category shape needs confirming against beat_contexts.json.
  4. parity-walk and edge-walk evals. Referenced by sections 7, 11, 12 but not among the 9 coach-behavior evals. Confirm the fleet has these two registered, or build them.
  5. Upstream gender routing owner. The rule says gender == woman selects this variant, but the code path that reads gender and picks the beatId is not spelled out in the source. The build must name it.
  6. create_your_own clip shared across variants. Both category and category-women bind the same clip on reveal-9. Fine for audio, but the link and audio checks must treat one clip bound from two beats as valid, not a duplicate error.
