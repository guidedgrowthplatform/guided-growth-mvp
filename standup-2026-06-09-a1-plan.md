# A1 ‚Äî Kill internal-narration phrases (in-depth plan)

> Goal: the coach never tells the user about its own internal operations. No "I'm saving that", no "Heading there now", no "Updating that habit". The user gives an answer ‚Üí the coach acknowledges briefly (or not at all, depending on screen) ‚Üí the coach asks the next thing. Tool calls run silently.

---

The agents found three independent sources of narration. Fixing the system prompt alone will NOT solve this ‚Äî the loudest source is a stack of hardcoded `requestStart` strings that Vapi reads out loud verbatim while a tool is running. The model isn't even involved when those play.

## Source 1 ‚Äî Hardcoded `requestStart` scripts (the biggest leak)

These are spoken by Vapi as filler audio while a tool round-trip is in flight. They are not generated by the LLM ‚Äî they are string literals on tool definitions. Every time the user provides info, Vapi reads one of these aloud. That's where "I'm saving that" comes from, literally.

**File:** `api/_lib/llm/tools.onboarding.ts`

| Line | Tool                    | Current `requestStart`                                            |
| ---- | ----------------------- | ----------------------------------------------------------------- |
| 122  | `submit_profile`        | "I'm saving that ‚Äî anything else about yourself you want to add?" |
| 179  | `submit_category`       | "Saving that ‚Äî does that one feel right?"                         |
| 201  | `submit_goals`          | "Got it ‚Äî want to add another goal, or are you good with that?"   |
| 289  | `update_habit`          | "Updating that habit ‚Äî anything else to tweak?"                   |
| 364  | `submit_custom_prompts` | "Saving your prompts ‚Äî want to add another, or are you set?"      |
| 415  | `navigate_next`         | "Heading there now."                                              |

### Options (need a decision)

- **Option A ‚Äî Delete them entirely.** Vapi falls back to silence while the tool runs. Cleanest but may create awkward dead air on slower tool calls (~500ms+).
- **Option B ‚Äî Replace with a single short neutral acknowledgement.** Something like "Mm." or "Okay." ‚Äî pure social token, no operational narration. Same on every tool, so it doesn't feel scripted.
- **Option C ‚Äî Replace with the next coaching question.** If we already know the next thing to ask, ask it during the requestStart slot. Risk: harder to author, can't be wrong about what's next.
- **Option D ‚Äî Use a single `requestStart` constant** and DRY them: one source, one place to tune.

**Recommendation:** **Option B + D** ‚Äî define `NEUTRAL_ACK = "Okay."` (or empty string after measuring tool latency) and use it everywhere. Cheapest, hardest to regress, easiest to undo.

## Source 2 ‚Äî Screen context examples teaching the narration style

The bundled screen contexts contain scripted "AI RESPONSE PATTERN" examples that include narration verbatim. The model sees these as in-context examples and copies the style.

**Concrete hits:**

- `src/generated/screen_contexts.json:30` ‚Äî VOICE-PREFERENCE: `"No problem. I'll write everything on screen."`
- `ai-qa/screens/checkins-morning.md:58` ‚Äî MCHECK-01: `"Got it ‚Äî all four logged."`
- `ai-qa/screens/checkins-morning.md:105` ‚Äî MCHECK-02: `"Got it ‚Äî [echo goal]"`
- `ai-qa/screens/pre-onboarding.md:198` ‚Äî VOICE-PREFERENCE: same as first

### Why this matters more than it looks

We can write the strongest "do not narrate" rule in the world, but if the screen context ‚Äî which is loaded INTO every prompt ‚Äî shows the model an example response that says "Got it ‚Äî all four logged", the example overrides the rule. Few-shot examples almost always win against instructions.

## Source 3 ‚Äî The Direct-LLM system prompt rule

Today's `NO_PRENARRATION_RULE` is narrow: it forbids pre-narrating the NEXT screen ("don't read out the next screen's opening line"). It does not forbid the model from narrating its CURRENT-screen tool calls. That's the gap.

**Composition order today** (`api/_lib/llm/buildSystemPrompt.ts:118`):

```
${coachingPreamble}${productBlock}

${NO_PRENARRATION_RULE}${onboardingNudge}${checkinNudge}${alreadyFilledBlock}${optionsBlock}${openerNudge}

${contextMessage}
```

### New rule ‚Äî draft

Create `api/_lib/llm/noInternalNarrationRule.ts`:

```ts
export const NO_INTERNAL_NARRATION_RULE = `## Don't Narrate Your Operations

The user is talking to a coach, not watching a machine work. Never describe what you are doing under the hood:
- Do NOT say you are saving, writing, recording, updating, adding, removing, or storing anything.
- Do NOT say you are opening, heading to, moving to, or navigating to anything.
- Do NOT say "one moment", "let me", "I'll", "I'm going to", "give me a sec", or any other phrase that announces your own next action.
- Do NOT confirm that a tool call worked, succeeded, or completed. The user does not know tools exist.

After the user answers, either say nothing operational at all and ask the next thing this screen needs, or give one short human acknowledgement ("Got it", "Makes sense", "Nice") and continue. Pick one ‚Äî never both.

If a tool call fails and you have the user's answer, retry the tool silently. Do not tell the user about the failure or the retry.`;
```

### Why this wording

- **Lists the failure modes verbatim** ("saving", "writing", "heading"). The narration-phrase audit showed the model is trained on these exact words via `requestStart` and screen-context examples. Naming them explicitly is the only thing that consistently moves model behavior.
- **Includes "I'll" and "I'm going to"** ‚Äî these are the soft form of narration ("I'll add that to your profile") that escape the obvious phrase ban.
- **One ack OR none, never both** ‚Äî closes the loophole where the model says "Got it" AND "I'm saving that". This pre-stages A2 (one ack per section).
- **Mentions tool failure silently** ‚Äî pre-stages A3 (silent retry on tool failure). Cheap to include now.

### Composition decision

Insert immediately AFTER `NO_PRENARRATION_RULE`:

```ts
${NO_PRENARRATION_RULE}\n\n${NO_INTERNAL_NARRATION_RULE}${onboardingNudge}${checkinNudge}...
```

Two blank lines between the two rules ‚Äî they are distinct concerns, side-by-side, not nested.

### Onboarding addendum cleanup

`api/_lib/llm/onboarding/systemPromptAddendum.ts:9` currently contains:

> "do not echo back, do not summarize ("got it, let me save that‚Ä¶")"

The parenthetical example is hostile here ‚Äî it primes the model. Either remove the parenthetical or replace with what to do instead. Recommendation: remove the parenthetical entirely; the new `NO_INTERNAL_NARRATION_RULE` is now carrying that load explicitly.

## Acceptance test ‚Äî how do we know A1 worked?

The current AI QA catalog (`scripts/ai-qa-verify.ts`) **only checks file/symbol existence**, not model output. It cannot test that the model stops saying these phrases. We need a separate output-level test.

### Proposal ‚Äî new test: `scripts/ai-qa-narration-test.ts`

A small Node script that:

1. Loads a fixture list of `(screen_id, user_message)` test cases
2. Calls `/api/llm` (local dev or against a deployed preview) for each
3. Streams the response back
4. Asserts that none of the forbidden phrases appear (case-insensitive)
5. Asserts no phrase matches the soft-form patterns (`/\bI'll\b/`, `/\blet me\b/`, `/\bI'm (saving|writing|adding|updating|opening|heading|moving)/`)
6. Returns exit code 1 if any case fails, with the failing screen + response

## Risks / things I'm not sure about

- **`requestStart` deletion may create dead air.** If tool calls take >1s round-trip, removing the filler audio could feel like the AI froze. Need latency data before committing to Option A. Option B ("Okay.") sidesteps this.
- **Few-shot examples in screen_contexts may dominate the prompt rule.** If we fix Source 3 but skip Source 2, the model will keep narrating because the context shows it doing so. Sources 2 + 3 must ship together to actually move the metric.
- **Vapi base prompt is out of our hands.** Path 1 (onboarding) will keep narrating until Yair updates the dashboard. Even with all of Sources 1‚Äì3 fixed, the most-visible part of the product (first-run onboarding) won't change without that. This is the bottleneck ‚Äî flag it early so Yair can move on it in parallel.
- **The catalog has no output assertions today.** Until `ai-qa-narration-test.ts` exists, we have no automated guard against regression. Worth doing as part of A1, not deferred.
