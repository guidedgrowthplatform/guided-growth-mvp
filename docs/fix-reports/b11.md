# B11 fix report: failing LLM calls (cancelled/empty responses, thinking stalls, habit-beat wedge)

Branch: `bugfix-loop3-llm` | MR: !401 | Date: 2026-07-02

## Symptom

Chat-mode LLM calls during onboarding failed repeatedly with a well-formed payload: the browser showed the request as cancelled/empty, the coach showed an indefinite thinking stall, and the flow wedged at the habit beats. Captured evidence: `b11-request-body.json` (screen ONBOARD-BEGINNER-04, mode chat, input voice).

## B18 rule-out

`origin/staging` tip (9526b377) builds clean: `npx tsc --noEmit` exit 0 and `npm run build` succeeds. (A first local failure was a stale shared node_modules missing the just-merged `@capacitor-community/apple-sign-in`; installing it locally resolved it. Not a repo build error.)

## Root causes (all verified in code, one verified live)

All three live in the shared Responses-API stream reader, `api/_lib/llm/openai-responses.ts` (`iterateEvents`), plus one output-budget mismatch in the route:

1. **`response.incomplete` was unhandled.** It is a terminal stream event (fires when `max_output_tokens` truncates the response). The reader ended silently, so the route in `api/llm/[...path].ts` treated the turn as a clean finish: no error frame, no response id, often zero text and zero complete tool calls. The client rendered an empty turn and the beat never advanced. Retrying hit the same wall, which is the wedge.

2. **Upstream provider errors were swallowed.** `response.failed` carries its detail under `event.response.error`, and a raw `error` event carries `code`/`message` at the top level. The old reader read `event.error` for both shapes, so every upstream failure collapsed to `openai_error: Unknown error`. The real provider error class never reached logs, the client, or the fix loop.

3. **Output budget too small for tool-heavy onboarding turns.** The route never overrode the reader's `max_output_tokens` default of 600. Onboarding turns emit tool JSON (add_habit, update_habit, advance_step, sometimes several per turn on the habit beats, and submit_brain_dump must echo the full transcript verbatim) on top of the coach's text. 600 output tokens is not enough for exactly the beats where B11 was reported.

4. **No server-side failure visibility.** `SENTRY_DSN` is not configured on the API functions (verified: the Sentry org contains only the `javascript-react` client project and zero server events), so `reportRequestFailure` no-ops. session_log end rows are best-effort and invisible to QA. Most failure classes left no trace anywhere a human looks.

Client-side amplifier: when the server sends an SSE `error` frame, `useLLM` aborts the fetch, which devtools renders as "(canceled)" with an empty body. That is why QA saw "cancelled/empty" even when the server did send an error frame. And a stream that died with no frame at all left `status: 'streaming'` forever (with the raw error codes rendered as coach text when a frame did arrive).

## Fix

Server (`api/_lib/llm/openai-responses.ts`, `api/llm/[...path].ts`):

- `iterateEvents` yields a new `incomplete` event for `response.incomplete` (with reason, response id, token usage); the route converts it into a retryable `incomplete_response` error frame instead of a silent empty finish. Partial tool call sets are never dispatched.
- `response.failed` and raw `error` events are parsed with their real shapes, so the upstream code/message survives to logs and the error frame.
- Onboarding requests set `max_output_tokens: 1000` (other screens keep 600).
- Every hard failure path logs one stable line: `[llm:fail] {stage, code, screen_id, mode, error_class, status, message}`. Grep Vercel function logs for `[llm:fail]`.
- A `x-gg-debug: 1` request header echoes `{ stage, class }` inside error payloads for QA. No stack traces to any client, flag or not.

Client (`src/hooks/useLLM.ts`, `src/hooks/useOnboardingChat.ts`):

- 30s idle watchdog on the SSE stream. Any event re-arms it; a dead stream aborts into a retryable `llm_timeout` error instead of an indefinite thinking stall. (Thinking-dot clearing itself is Loop 4's B19 fix on `bugfix-loop4-timeline`; not duplicated here.)
- One silent automatic retry per turn for transient classes (timeout, truncation, upstream 5xx/provider error, killed stream), then ONE friendly bubble: "Something didn't work on my end. Mind saying that again?". Raw error codes no longer render as coach text.
- A failed opener degrades to the beat's authored line plus its card (via the `sendOpener` result), so a beat can no longer render as dead air with its card lost.

## Upstream prevention

- The stream reader now handles every terminal event type of the Responses API; a future unknown terminal still ends the round visibly through the existing exception path rather than silently.
- Any new failure path added to the llm route should call `logFailure(stage, code, err)`; the stable prefix makes regressions greppable on any deploy.
- When adding beats whose tools carry large payloads (verbatim transcripts, multi-habit writes), budget output tokens explicitly; the 600 default is for plain chat.
- Setting `SENTRY_DSN` on the Vercel API functions would turn `reportRequestFailure`/`reportToolFailure` from no-ops into real telemetry. Not done in this MR (env change, Yair's call).
