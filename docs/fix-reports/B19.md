# B19: a loading bubble renders or sticks on some beats when the beat loads

Status: FIXED
Branch: bugfix-loop4-timeline

## Symptom

On some beats a coach "thinking" loading bubble (ThinkingDots) appeared the
moment the beat loaded, before the user did anything; sometimes it stayed
stuck on the beat.

## Root cause

The active beat's CoachThinkingIndicator (BeatPlayer.tsx useCoachThinking)
rendered dots when `(awaiting || chatBusy) && !speaking`. Two defects:

1. Render-on-load: `chatBusy` is the GLOBAL Direct-LLM stream flag. When a
   beat advances while the previous beat's stream is still in flight (the
   common case: the data tool that advanced the beat is part of that stream),
   the NEW beat's indicator mounts with `chatBusy` already true and shows a
   loading bubble at beat load, with no user turn on that beat.
2. Stuck: `awaiting` latches on a user final and clears only on assistant
   transcript activity or speech. A stream that settles with NO assistant
   output (LLM error, empty reply, cancelled call, the B11 class) never clears
   it, so the dots stick indefinitely.

## Fix

Extracted the logic into a pure reducer,
src/onboarding-flow/renderer/coachThinking.ts:

- Only a chatBusy RISE observed while this beat's indicator is mounted counts
  (`busy-rose`); a stream inherited from the previous beat never lights the
  dots on the new beat.
- A busy stream settling (`busy-settled`) clears both latches, so a reply-less
  failure can no longer strand the bubble. On the Vapi path chatBusy never
  rises, so awaiting still clears only on assistant activity, unchanged.

The hook (useCoachThinking) is now a thin adapter: transcript bus events plus
chatBusy edge detection dispatched into the reducer.

## Verified

- Pure reducer unit tests cover: fresh mount dark, inherited-busy dark,
  user-final lights, assistant clears, reply-less settle clears (the stuck
  case), speaking suppresses, Vapi path preserved.
- Preview smoke: no thinking dots at beat load across the beginner-lane walk
  (state-check spot check plus the full tap-through walk).
- The live failing-LLM stick could not be mechanically reproduced locally (no
  LLM backend in the vite preview); the settle-clears rule is proven at the
  reducer level and should be re-checked once on a preview deploy alongside
  Loop 3's B11 work.

## Upstream prevention

The rules are in a pure module with tests; any future indicator behavior
change must go through the reducer, and a regression to global-busy or a
non-clearing settle fails the suite.
