# B21: completed subcategory beat removed, then habit-selection and habit-schedule render simultaneously

Status: FIXED (three mechanisms closed; one root cause carries a Loop 2 dependency, see Open items)
Branch: bugfix-loop4-timeline

## Symptom (live QA walkthrough 2026-07-02)

After answering the subcategory (goals) beat, that completed beat disappeared
from the timeline, then habit-selection and habit-schedule rendered at the same
time instead of selection first, schedule after a habit is chosen.

## Mechanisms found and fixed

1. The removal is the B6/B7 unmount class (as the ledger predicted). A
   completed goals beat whose thread held user turns but no committed coach
   opener lost its coach line and frozen card, so it read as REMOVED. Fixed in
   the B6/B7 commit (past beats always render their card and replay the
   authored opener). See docs/fix-reports/B6.md and B7.md.

2. QA startAt burn-through (REPRODUCED, fixed). `fastForwardToNode` walked
   branch nodes with an empty capture; `answers.path` never resolved, so the
   fork fell through to its MERGE node and the walk ran the machine to
   COMPLETION. `?startAt=goals` (or any lane node, including the QA control
   screen's one-tap tiles) rendered the ENTIRE flow at once as past beats,
   with habit-select and habit-schedule visible simultaneously and the lane
   beats (category, goals) missing entirely. Fix: the fast-forward now
   resolves a branch TOWARD the lane containing the target
   (laneValueContaining), and an unreachable target returns the START state
   (as the doc comment always claimed) instead of a burned-through flow.

3. Leading-edge double-advance without data (defensive gate). The coach-save
   advance effect moved the engine forward on any server current_step climb
   past the active beat's step. Two step events in quick succession (e.g. a
   submit write then a chained advance write, or a server-side map still on
   the stale V2 scale, Loop 2's item) advanced habit-select with an EMPTY
   habitConfigs capture: an empty frozen picker appeared next to the active
   schedule card, i.e. both "simultaneously". New pure gate
   `captureCompletesBeat`: a beat that persists fields only advances off a
   server climb when the persisted data actually replays its capture (or the
   fork's path). Persist-less beats (why-intro, into-app, projections) advance
   freely; the state-check proxy behavior is untouched (Loop 2 territory).

Also in this commit, QA observability used to reproduce all of this: the flow
renderer tags every beat with `data-beat-id` / `data-beat-active` (a
display:contents wrapper, zero layout impact), and the auth-free preview route
(/onboarding-flow-preview) supports `?startAt=<nodeId>` like the real route.

## Verified

- Preview: `?startAt=goals` lands ACTIVE on goals with the lane walked
  (category past, fork resolved to simple) and NO habit beats rendered; before
  the fix it rendered all 15 non-lane beats at once, machine complete.
- Full tap-through walk (profile to into-app): visited is append-only, exactly
  one `data-beat-active` beat at every dump, habit-select renders alone until
  answered, then habit-schedule alone.
- New tests (src/onboarding-flow/__tests__/timelineStability.test.ts): a
  scripted full run asserts the completed-beat timeline only grows and the
  active beat is always the single tail; fastForwardToNode lane resolution +
  unreachable-target identity; captureCompletesBeat blocks the empty
  habit-select skip and passes every legitimate advance shape.

## Open items for the next verification pass

- The server-side step-write shape (which values the tool handlers write per
  save) is Loop 2/3 territory: after Loop 2's map fixes land, re-run the live
  voice walkthrough on a preview deploy and confirm the coach-driven advance
  through goals, habit-select, habit-schedule stays sequential.
- The data gate means a coach that advances current_step WITHOUT persisting
  the beat's fields no longer drags the UI forward; if a legit tool does that
  anywhere, it must persist its fields in the same write (that is the
  contract).
