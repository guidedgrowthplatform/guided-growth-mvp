# B7: profile card disappears after age and gender are given

Status: FIXED
Branch: bugfix-loop4-timeline

## Symptom

Once the profile beat completed (age + gender given), its card, and often its
coach line, vanished from the scrollback. Only the user's answer bubble stayed.

## Root cause

Past-beat rendering split on `hasBeatConversation` (any stored turn for the
beat's screenId). When true, BeatView rendered `BeatConversation active={false}`
whose card reveal was gated on `openerPresent`, which requires a COMMITTED
coach turn in the message store positioned before any user turn.

On the v3 Direct-LLM beats the opener is never committed to the store: BeatView
karaokes the authored line locally (BeatPlayer), while the user's spoken answers
and the coach's replies ARE committed. So a completed beat's thread starts with
a user turn: `openerIdx` resolves to none, `openerPresent` stays false, and the
frozen card never rendered. The coach line disappeared for the same reason
(nothing to render for a null opener, and no fallback was passed to past
beats).

## Fix (src/onboarding-flow/renderer/BeatPlayer.tsx + BeatView.tsx)

1. A PAST beat's card always renders: `shouldRevealCard` is unconditionally
   true when `active` is false. Reveal gating is an active-beat concern; a
   completed beat is a persisted receipt.
2. BeatView passes the authored opener to the past-beat BeatConversation as
   `fallbackOpener`; when the thread has no committed opener, the authored
   line(s) render statically above the card (one bubble per turn line, sharing
   the B5 openerTurns convention). A committed opener still wins (no
   duplication).

The active-beat behavior is untouched: the currently-active card still updates
in place, and active reveal gating (karaoke, warm opener, 12s failsafe) is
unchanged. FlowCheckinPreview and the home tour share this renderer; their past
beats carry no frozen card (card undefined), so the card change is a no-op
there and the authored-line fallback only ADDS the coach line that previously
vanished.

## Verified

- Preview repro (seeded thread in the live-run shape: user STT turn + Direct-LLM
  reply, no committed opener; localStorage `mvp03_onboarding_thread_*`): past
  profile beat now renders the three authored opener turns, the frozen profile
  card, the user bubble, and the coach reply. Before the fix it rendered only
  the dialogue bubbles.
- Full tap-through walk (startAt=profile to into-app) stays append-only.
- tsc clean; onboarding-flow + contexts suites green.

## Upstream prevention

The rule is now structural: past-beat cards are never gated on transcript
shape. The B6 report covers the sibling never-disappear rule for bubbles; the
timeline stability test added with B21 asserts completed beats only accumulate.
