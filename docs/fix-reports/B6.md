# B6: chat bubbles disappear randomly during a run

Status: FIXED (two verified vectors closed)
Branch: bugfix-loop4-timeline

## Symptom

During an onboarding run, bubbles that were on screen disappeared later:
sometimes a completed beat's coach line, sometimes previously rendered turns.

## Root causes (two distinct vectors, same visible symptom)

1. Past-beat branch flip dropping the coach line and card. A completed beat
   rendered fine while `hasBeatConversation` was false (authored opener +
   frozen card branch). The moment ANY turn for that screenId landed in the
   message store (an STT final committing late, hydration seeding a row), the
   beat flipped to the BeatConversation branch, which rendered ONLY committed
   turns: no committed opener means no coach line and (via the openerPresent
   card gate) no card. Bubbles visibly vanished at the flip. Shared root cause
   with B7, fixed together: past beats always render their card, and the
   authored opener replays when the thread has no committed coach turn.

2. Cross-device hydration REPLACING the live thread. In
   OnboardingVoiceProvider, `fetchOnboardingThread` seeding used
   `cur.length >= seeded.length ? cur : seeded`: whenever the server set was
   longer than the in-memory one, the live thread was replaced wholesale.
   Live turns that had not mirrored to the server yet were silently dropped,
   which reads as "bubbles disappear randomly" a few seconds into a run
   (whenever the fetch resolves). Fixed with `mergeThreadMessages` (new pure
   module src/contexts/onboardingThreadMerge.ts): server history is the base
   order, a shared id keeps the fresher (longer) live copy, and unmirrored
   live turns append at the tail. Nothing on screen is ever removed.

Not touched, by design: `shouldWipeOnAnonIdChange` (the anon-id-change wipe
gate protecting real users) and the localStorage hydrate, which already only
fills an EMPTY thread.

## Verified

- Preview repro with a seeded thread (see B7.md): the branch flip now renders
  the same content in both branches, so a late-landing turn adds bubbles
  instead of removing them.
- mergeThreadMessages unit tests: live-only turns survive a longer server set
  (the exact drop case), fresher live text wins, no duplicate ids.
- Full tap-through preview walk: no bubble count ever decreased across the
  whole beginner lane.

## Upstream prevention

The merge is a pure, unit-tested function; any future hydration source should
route through it. The append-only timeline test (added with B21) locks the
machine-level guarantee; the past-beat render rules (B7) lock the view-level
one.
