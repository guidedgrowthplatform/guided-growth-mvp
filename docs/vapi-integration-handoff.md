## 1. Beat-context system (the heart of the coach)

Onboarding coach copy is two layers, both sent fresh every turn:

- **Global context** — who the coach is, the Path 1/2/3 behaviour, the cross-beat
  rules. Sits above every beat.
- **Per-beat context** — the one thing to collect on this beat, the verbatim
  opener line, and (code-owned) which tools are allowed.

Source of truth is the App Master Sheet **"Beats Context"** tab. `allowedTools` is
**never** stored in the Sheet/DB — it's code-owned in `beatContexts.ts` and stays
there permanently (it's the safety gate, not content).

## 2. Supabase sync (Sheet → DB → repo → runtime)

A one-way publishing pipeline so coach copy can be edited outside code. It was
**built but never stood up** before; it is now **live**.

```
 Master Sheet "Beats Context" tab
        │  sync_beats_context.py   (Sheet → Supabase)
        ▼
 Supabase: beat_contexts (15 rows) + onboarding_globals (1 row)   ← migration 052
        │  sync_beat_contexts.py   (Supabase → repo)
        ▼
 api/_lib/llm/onboarding/beatContexts.generated.json
        │  overlaid at import onto hand-authored defaults
        ▼
 api/_lib/llm/onboarding/beatContexts.ts   (Direct-LLM reads this)
```

What was done:

- **Migration 052** (`beat_contexts` + `onboarding_globals`) was **run on prod**
  (the tables did not previously exist).
- **New durable `scripts/voice-sync/sync_beats_context.py`** reads the live
  "Beats Context" tab (Google service account, same as `seed_contexts.py`) or an
  offline snapshot (`--from-json`, no Google creds) and upserts the 15 onboarding
  beats + the onboarding global into Supabase, with FNV-1a content hashing +
  version bumps. This replaces the old `seed_beat_contexts.py`, which hand-copied
  an inline block that drifted.
- **`sync_beat_contexts.py`** (Supabase → repo) generated `beatContexts.generated.json`,
  so the synced copy is now overlaid onto the code at import. **15 beats + the
  onboarding global (≈3.9k chars) are live.**
- The Cartesia/check-in `GLOBAL` row in the Sheet is a separate surface with no
  consumer yet — parked.

Regen chain when the Sheet changes:
`sync_beats_context.py` → `sync_beat_contexts.py` → `npm run beats:bundle` (below).

## 3. Unification — ONE source feeds every path

Previously two parallel context pipelines diverged: Direct-LLM read `beatContexts.ts`;
Vapi read a separate `screen_contexts.json` bundle (with no global) and the Vapi
dashboard held yet another copy of the global. They drifted.

Now **Vapi is brought onto the same synced source Direct-LLM uses.**

- **Frontend beat bundle** `src/generated/beat_contexts.json` (generated by
  `scripts/build-beat-bundle.ts`, `npm run beats:bundle`): the synced coach copy +
  opener + code-owned `allowedTools` + the per-beat step/target — so the frontend
  (Vapi) can read the same content the backend (Direct-LLM) uses.
- **Machinery composer** `src/lib/context/onboardingBeatBundle.ts`: Vapi gates
  tools by _prose_ (it has no structural per-beat tool filter), so the context it
  receives must still carry the machinery — `ALLOWED`/`FORBIDDEN` tools +
  `navigate_next(target_step=N)`. That machinery is **generated from code** (the
  engine step model), then prepended to the clean **synced coach copy**.
- **`getScreenContext`** now serves `ONBOARD-*` from this composed block for both
  Vapi cold-start and the mid-call heartbeat. Non-onboarding screens still use the
  old `screen_contexts` bundle.

Result: **machinery follows the engine; copy follows the Sheet.** One Sheet edit
flows to every path; no more dashboard-vs-code global drift on the per-beat copy.

## 4. Visited-beat / step order (derived from the flow builder)

The agent aligned the navigation machinery to the **flow builder** as the order
authority (`src/onboarding-flow/flows/onboarding-beginner-v1.generated.json`),
rather than a hand-maintained step map.

- Each node carries `nextId`/`backId` (linked-list order) + `persist.step`. The
  effective step is `persist.step ?? ENGINE_PERSISTLESS_STEP[screenId]` (mirrors
  the orchestrator), e.g. plan-review = step 6.
- The Vapi `navigate_next(target_step)` for a beat = the **next node's step,
  skipping same-step nodes**. So **habit-select (03) and habit-schedule (04) are
  ONE step (5)** and both target plan-review (6) — "habits is one beat: add, then
  configure one by one."
- Tail follows the flow builder: plan-review(6) → morning-setup(7) →
  reflection(8) → complete (terminal, `confirm_plan`). The `ONBOARD-COMPLETE`
  beat is terminal: `confirm_plan` only, `navigate_next` forbidden.
- A read-only diff (`scripts/diff-beat-machinery.ts`) confirms the generated
  machinery reproduces the working targets and only differs where the flow
  builder is intentionally authoritative.

Tests lock this: `src/lib/context/onboardingBeatBundle.test.ts`.

## 5. Vapi coverage — the whole flow is live

`CHAT_VAPI_BEAT_SCREENS` was widened from {profile, fork, category, goals,
habit-select} to **every onboarding beat** (adds habit-schedule, advanced,
plan-review, morning-setup, reflection, complete). The step-model + bundle gaps
that held the tail back are closed by the unified bundle, so Vapi now runs **one
continuous live session across the whole flow** instead of going text-only past
habits. AUTH and MIC stay silent (no coach turn).

## 6. Live chat experience

The flow chat now behaves like a live conversation, not a pre-rendered script.

- **One chronological feed.** The active beat renders from the provider's single
  `messages` store (every user + coach final, in order, merged per turn, tagged by
  beat) + the live partial of the in-progress turn. This fixed an ordering bug
  where user and coach turns were drawn from separate buffers and interleaved
  wrong (a reply could appear above the line it answered).
- **Live user + coach turns** render word-by-word; one bubble per uninterrupted
  utterance.
- **Typing indicator** (neutral dots) after the user finishes a turn until the
  coach replies / while a tool is in flight. No words (privacy: never narrate the
  system).
- **Rigid verbatim opener.** Cold-start: Cartesia speaks the verbatim opener while
  Vapi connects (latency mask). Warm beats: `pushScreenContext` prepends a
  verbatim-opener directive so Vapi opens each beat with the authored line
  word-for-word, never improvised — skipped for the one beat Cartesia handled so
  it never doubles.
- **Persistence across refresh.** The thread is persisted to localStorage per
  anon_id (`onboardingThreadStore`): hydrate on load, save on every turn, wipe
  only on a real user switch. The conversation survives a reload.
- **Frozen past-beat receipt cards.** Completed beats re-render their card frozen
  in its captured state (`componentRegistry` `readOnly`/`frozen`) so the scroll
  reads back as a real conversation of filled receipts.
