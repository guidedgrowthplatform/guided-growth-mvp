# Onboarding Acceptance Deepening

This is a proposed replacement depth standard for the consolidated onboarding render. A criterion passes only when its stated observation is true; statements that merely restate a beat's purpose are not acceptance criteria.

## Global interpretation

- **Per-beat gate:** each section states its source-declared advance condition. Refusal, silence, invalid input, incomplete input, or tool failure blocks advance only where that beat's source says it does; structural, auth, and OS-permission beats use their own gates.
- **Locked-copy oracle:** where the source marks copy verbatim or supplies an exact edge string, the rendered English words must equal that source text after slot substitution and playback must use the cited clip; punctuation, audio bytes, TTS prosody, and other locales are not compared unless the source separately locks them. Empty script lines are reveal events, not permission for coach speech.
- **Tool boundary:** the check records every call and fails a beat on any tool outside that beat's effective allowed set. Save-before-capture and advance-before-save checks apply only to beats whose source declares those operations.
- **Persistence boundary:** DB probes assert only the exact `dataOut` keys and destinations quoted from `beatsSource.ts`; OS permission state is checked through the OS stub, not as a DB field.
- **Check methods:** `Harness` identifies behavior suitable for an existing or future event/tool test; `DB probe` identifies a before/after read of the cited destination; `Manual` identifies OS or perceptual behavior. These are test specifications, not claims that this document executed them, and they do not require a new fixture framework.
- **Source conflicts:** prefer an entry's explicit `AUTHORITATIVE RENDER CONTRACT`, then its Bible rules, then top-level metadata. A conflict is not silently merged: the losing behavior is named in that beat and prohibited until the source owner reconciles the entry.
- **Variations:** every concrete source ID is mapped below. It inherits its section only when tools, data, cap, gate, and branch behavior match; variation-specific copy/options are always checked against that entry.
